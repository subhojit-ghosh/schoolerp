# RBAC — Full Plan

## Goal

Replace the current coarse `activeContext === "staff"` authorization with a
complete role-based access control system. When done, every backend operation
requires a specific named permission, roles carry fixed or configurable permission
sets, institution admins can manage custom roles through the UI, staff members
have role assignments with optional scopes, and data queries are filtered to what
each role is allowed to see.

---

## Mental Model

```
User
 └── Member (staff | guardian | student)
      └── MembershipRole (which role, valid date range)
           ├── Role (set of permissions)
           │    └── RolePermissions (permission slugs)
           └── MembershipRoleScope (campus | class | section)
```

Every API call checks in order:
1. Valid session? — `SessionAuthGuard`
2. Correct tenant? — `TenantInstitutionGuard`
3. Has the required permission? — `PermissionGuard` (Phase 1)
4. Is the target data within their scope? — query-level filtering (Phase 2)

## Role Model

### System roles (platform-defined, fixed)

| Role slug | Display name | Who is this |
|---|---|---|
| `institution_admin` | Institution Admin | Principal / trust head — full access including settings |
| `school_admin` | School Admin | Office admin — full ERP, no institution settings |
| `academic_coordinator` | Academic Coordinator | Manages academic structure and marks |
| `finance_manager` | Finance Manager | Manages fees only |
| `class_teacher` | Class Teacher | Full access scoped to their assigned class/section |
| `subject_teacher` | Subject Teacher | Attendance and marks scoped to their sections |

System roles: `isSystem: true`, `isConfigurable: false`. Cannot be deleted or
renamed by institution admins.

### Custom roles (institution-defined)

Institution admins can create additional roles (e.g. "Librarian", "Counsellor"),
pick permissions from the platform permission set, and assign them to staff.
Custom roles: `isSystem: false`, `isConfigurable: true`. Deletable when no active
assignments exist.

## Scope Model

Scopes limit what data a role assignment applies to. A `class_teacher` with scope
`section: 8B` can only read/write attendance and marks for students in 8B.

Scope types (narrowing order): `institution → campus → class → section`

A role assignment with no scope = institution-wide access for that role.
A role assignment with `campus` scope = access limited to that campus.

Scope enforcement is at the **query level** — service methods filter results using
the user's resolved scopes. It is not just a frontend visibility concern.

## Phase 2 — Scope Enforcement (backend only)

**Outcome:** Data queries respect the scope of a user's role assignment. A
`class_teacher` scoped to Section 8B only sees 8B students, attendance, and marks.
A campus-scoped `school_admin` only sees data belonging to their campus.

**Does not include:** UI for assigning scopes (done in Phase 4).

### What changes

`AuthService` gains a `resolveScopes()` method that returns the user's scopes for
a given institution:

```ts
type ResolvedScopes = {
  campusIds: string[] | "all";
  classIds: string[] | "all";
  sectionIds: string[] | "all";
};
```

`"all"` means no scope restriction (institution-wide role). An array means
restrict to those IDs only.

### Steps

- [ ] **Step 1 — `AuthService.resolveScopes()`**

  Join `member → membershipRoles → membershipRoleScopes` for the user in the
  institution. Aggregate scope IDs by scope type. Return structured result.

- [ ] **Step 2 — Scope-aware query helpers**

  Add helpers to domain services (or a shared `ScopeFilter` utility) that accept
  resolved scopes and apply `inArray(table.campusId, scopedCampusIds)` /
  `inArray(table.sectionId, scopedSectionIds)` where relevant.

- [ ] **Step 3 — Apply scope filtering to each domain service**

  Resolve scopes at the top of each list/detail service method and pass to query.

  | Service | Scope column |
  |---|---|
  | StudentsService — list | `campus_id`, `section_id` (via enrollment) |
  | StudentsService — detail | verify student belongs to scoped campus |
  | AttendanceService — list/write | `section_id` |
  | ExamsService — marks list/write | `section_id` |
  | ClassesService — list | `campus_id` |
  | GuardiansService — list | via linked student campus/section |
  | FeesService — list | `campus_id` |
  | StaffService — list | `primary_campus_id` |

- [ ] **Step 4 — Scope bypass for institution-wide roles**

  When `resolveScopes()` returns `"all"` for a dimension, skip that filter
  entirely. Institution admin and school admin always get `"all"`.

### Phase 2 — Definition of Done

- [ ] `resolveScopes()` returns correct scopes for campus-scoped and section-scoped roles
- [ ] Institution-wide roles bypass all scope filters
- [ ] `class_teacher` scoped to section 8B cannot read students from section 9A
- [ ] `finance_manager` scoped to Campus A cannot read Campus B fee data
- [ ] Scope filtering does not affect performance for institution-wide roles
- [ ] Typecheck passes
- [ ] OpenAPI export succeeds

## Phase 3 — Role Management UI

**Outcome:** Institution admins can view system roles, create custom roles, assign
permissions to custom roles, and delete empty custom roles — all through the ERP
settings UI.

### Backend steps

- [ ] **`GET /roles`** — list roles for the institution (system + custom)
- [ ] **`GET /roles/:id`** — role detail with assigned permissions
- [ ] **`POST /roles`** — create custom role (`institution:roles:manage` required)
- [ ] **`PATCH /roles/:id`** — update name/permissions of custom role
- [ ] **`DELETE /roles/:id`** — delete custom role; 409 if active assignments exist
- [ ] **`GET /permissions`** — list all platform permission slugs (for picker UI)

All role mutation endpoints require `institution:roles:manage` permission.

### Frontend steps

- [ ] **Settings → Roles** — list page showing system roles (read-only badges) and
  custom roles (editable)
- [ ] **New role sheet/page** — name input + permission picker (grouped by resource)
- [ ] **Edit role sheet/page** — same form, pre-filled; system roles show permissions
  as read-only
- [ ] **Delete role** — row action; disabled with tooltip when role has active assignments

### Phase 3 — Definition of Done

- [ ] Institution admin can list all roles
- [ ] System roles display as read-only
- [ ] Custom role can be created with selected permissions
- [ ] Custom role can be edited
- [ ] Custom role deletion blocked when active assignments exist
- [ ] `institution:roles:manage` permission enforced on all mutation endpoints
- [ ] Typecheck passes
- [ ] OpenAPI regenerated and ERP API types updated

## Phase 4 — Staff Role Assignment UI

**Outcome:** Institution admins can assign roles to staff members and optionally
scope those assignments to a campus, class, or section. The staff detail page
shows current role assignments and allows adding or removing them.

### Backend steps

- [ ] **`GET /staff/:id/roles`** — list current role assignments for a staff member
- [ ] **`POST /staff/:id/roles`** — assign a role with optional scope
  (`institution:users:manage` required)
- [ ] **`DELETE /staff/:id/roles/:assignmentId`** — remove a role assignment
- [ ] Validate scope consistency on assignment:
  - `section` scope requires `class` scope to be in the same campus
  - Cannot assign a scope that the assigning admin cannot themselves see

### Frontend steps

- [ ] **Staff detail page → Roles section** — list current assignments with scope
  labels (e.g. "Class Teacher — Campus A / Class 8 / Section B")
- [ ] **Assign role sheet** — role picker dropdown (filtered to institution's roles)
  + optional scope cascade: campus → class → section
- [ ] **Remove assignment** — row action with confirmation

### Scope cascade UI

The scope picker is a three-step conditional cascade:
1. Campus — always shown; optional
2. Class — shown after campus is selected; filtered to selected campus
3. Section — shown after class is selected; filtered to selected class

Selecting a narrower scope without a broader one is not allowed
(e.g. cannot pick a section without first picking its campus and class).

### Phase 4 — Definition of Done

- [ ] Staff member can be assigned one or more roles
- [ ] Role assignment can have campus, class, or section scope
- [ ] Scope cascade works correctly in the UI
- [ ] `institution:users:manage` enforced on assignment endpoints
- [ ] Removing an assignment is reflected immediately in the staff detail page
- [ ] A user with multiple roles accumulates permissions from all of them
- [ ] Typecheck passes
- [ ] OpenAPI regenerated and ERP API types updated

## What This Plan Does Not Cover

- Parent and student context authorization (different model — guardian and student
  reads are scoped by relationship, not by roles)
- Platform-level admin (SaaS operator) access
- Time-bound role assignments with `validFrom` / `validTo` enforcement in the UI
- Department-level scopes (deferred — no department module yet)
- Audit logging for role assignment changes
